# TypeWhale 产品需求文档（PRD）

最后更新：2026-06-30
当前本地安装版基线：1.6.6 (Build 456)
文档性质：产品级需求、边界、验收与路线图；不是开发日志，也不是逐条 changelog。

## 1. 产品定义

TypeWhale 是一个本地优先的桌面语音输入与智能文字处理工具。它让用户在任何当前工作流中，通过全局快捷键完成“说话 -> 本地识别 -> 可选智能整理/翻译 -> 自动写回目标应用”的输入链路，并通过一个不抢焦点的录音胶囊提供实时确定感。

TypeWhale 的核心不是“语音转文字 demo”，而是一个可长期后台运行、可商业化分发、能进入用户真实工作流的 macOS 输入生产力工具。

一句话价值：

> 让高频桌面用户用语音稳定、低打扰、可控地完成原本需要键盘输入、复制、整理和翻译的文字工作。

## 3. 产品原则

### 3.1 本地优先

默认语音识别在本机完成。音频、识别文本、剪贴板内容和目标应用上下文都属于敏感数据。任何新增网络发送能力都必须在代码、UI 和文档中显式说明。

当前允许的远程能力：

- DeepSeek 智能整理。
- DeepSeek 自动翻译。
- DeepSeek 截图翻译。

这些能力必须是用户可理解、可配置、失败可回退的增强层，不能成为本地输入主链路的硬依赖。

### 3.2 输入链路优先

语音输入主链路的成功标准高于功能丰富度：

```text
全局快捷键 -> 录音 -> 胶囊反馈 -> final 本地识别 -> 粘贴回目标应用
```

任何新功能都不能破坏：

- 快捷键响应。
- 录音稳定。
- 胶囊确定感。
- final 识别质量。
- 目标应用粘贴正确性。
- 剪贴板恢复。

### 3.3 胶囊是信心锚点，不是装饰

录音胶囊的作用是让用户在说话时确认“应用正在听、正在理解、没有失控”。它必须安静、快速、稳定、不抢焦点。

胶囊实时预览不是最终文本来源。最终粘贴必须来自完整录音的 final ASR。

### 3.4 已验证好体验不能被“修 bug”顺手砍掉

历史上 1.2 胶囊体验被用户确认更好。后续所有胶囊、预览、动效、停顿、分块、VAD 改动，都必须尊重这个事实：

- 自然停顿不能让预览窗口重置。
- 已显示的稳定前缀不能反复跳变。
- 不能为了避免闪烁而删除运动连续性。
- 不能为了性能退回上下文断裂的短窗口识别。

### 3.5 后台运行必须低打扰、可恢复

TypeWhale 是状态栏常驻工具，不是每次使用后退出的单次工具。它必须处理长期后台、睡眠唤醒、权限变化、模型内存增长和目标应用变化。

内存治理的产品目标是“持续响应快，同时避免异常膨胀”，不是“尽可能省内存”。ASR/VAD 模型不应在每次输入后卸载，也不应因空闲定时卸载导致下一次开口卡顿。高内存安全网使用动态预警阈值：`min(20GB, max(2GB, 本机物理内存 * 25%))`，避免回到 1GB 附近频繁 flush/reload。

### 3.6 截图是观察工具，不能改变桌面状态

进入截图模式不应唤醒 TypeWhale 主窗口，也不应改变普通 app 的 z-order。只有用户明确选择某个窗口截图时，才可以为了捕获目标窗口而临时置顶该窗口并重新截图。

## 4. 目标用户与场景

### 4.1 高频文字输入用户

代表场景：

- 微信、飞书、邮件、备忘录、浏览器输入框中快速输入。
- 不想离开当前上下文打开独立转写工具。
- 希望录完后文本直接出现在原输入位置。

核心诉求：

- 快捷键一按就能说。
- 不打断当前应用。
- 粘贴目标准确。
- 识别失败时结果不丢。

### 4.2 产品负责人 / 开发者 / AI 协作用户

代表场景：

- 口述开发需求、bug 反馈、复盘和任务拆解。
- 在 IDE、Codex、Obsidian、聊天工具之间频繁切换。
- 需要把口语整理成更适合执行的需求或行动项。

核心诉求：

- 保留原始语义、约束和情绪强度。
- 自动识别开发场景并选择开发需求整理。
- 不把用户口述内容误当成给模型的新指令。
- 最近转录和需求池能兜住失败或后续整理。

### 4.3 双语沟通用户

代表场景：

- 中文口述，自动转成自然英文聊天/工作表达。
- 英文内容截图后翻译成中文并保留在图片中。
- 需要区分“润色”和“翻译”，避免智能整理擅自改变语言。

核心诉求：

- 翻译开关独立。
- 翻译方向明确。
- 译文自然，不机械。
- 成本和失败状态可见。

### 4.4 长时间后台使用用户

代表场景：

- TypeWhale 一直开着，白天断续使用。
- 电脑睡眠、唤醒、接入蓝牙耳机、通话软件切换输入设备。
- 期望它像输入法一样稳定，而不是像实验脚本一样需要重启。

核心诉求：

- 空闲不发热。
- 下一次开口不冷启动。
- 睡眠唤醒后快捷键恢复。
- 权限和模型状态可诊断。

### 4.5 Windows 平台潜在用户

Windows 版当前不是 macOS 完整能力的同级产品，而是平台扩展 MVP。

第一阶段目标：

```text
全局快捷键录音 -> Silero VAD -> SenseVoice final 本地识别 -> Ctrl+V 粘贴
```

非第一阶段目标：

- macOS 胶囊完整动效。
- 实时预览。
- 截图翻译。
- 智能整理。
- 托盘、安装包和签名之外的商业化完整体验。

## 5. 产品范围

### 5.1 当前主线能力

P0 当前能力：

- macOS 原生 App。
- 状态栏常驻入口。
- 全局录音快捷键。
- 单击切换录音、长按说话。
- 非激活录音胶囊。
- 本地 SenseVoice / sherpa-onnx final ASR。
- Silero VAD 人声判定。
- 自动粘贴到目标应用。
- 剪贴板恢复。
- 最近转录历史。
- 权限、模型、热键和运行状态诊断。

P1 当前能力：

- 胶囊实时预览。
- 胶囊 dBFS 输入电平。
- 停顿自动完成。
- 录音时降低系统音量。
- Apple Voice Processing 麦克风语音增强。
- 智能整理。
- 自动翻译。
- 自定义提示词、开发术语和自动规则。
- DeepSeek 成本统计和余额展示。

P2 当前能力：

- 截图选区、窗口候选、标注、OCR、复制和保存。
- 英文截图翻译为中文。
- 行级 OCR 翻译回贴。
- 本地需求池。
- Windows WPF MVP 源码和交接计划。

### 5.2 当前非目标

当前不承诺：

- 作为系统输入法安装。
- 替代 macOS dictation。
- 完整离线智能整理。
- 云端语音识别。
- 自动上传音频或遥测。
- 公证后的正式公开商业分发包。
- Windows 与 macOS 完全同功能。
- 中英自动判断的截图双向翻译。
- 生成式图片背景修复。

### 5.3 已搁置或回滚方向

这些方向不应被后续 agent 当成“待恢复默认方案”：

- macOS 12 兼容：已放弃，当前基线为 macOS 14+。
- Python worker：已移除，当前为原生 sherpa-onnx / ONNX Runtime bridge。
- 每次输入后卸载 ASR/VAD：已否决，会破坏连续输入速度。
- 空闲 N 秒后卸载模型：已取消，下一次输入不能冷启动。
- 能量阈值 VAD 作为人声判定：已删除，仅保留能量用于波形和 dB 读数。
- 近场幅度门（按音量幅度判断远/近场再决定是否进预览）：已尝试并移除，仅靠幅度无法区分远近，且容易误挡用户自己说话。
- 停顿后重置实时预览窗口：已回滚，破坏自然口述体验。
- WebRTC VAD 前置粗筛：已搁置，当前性价比不足。
- 预览短分段拼接路线：历史上造成跳变和上下文断裂，不得直接复活。
- 截图双向翻译：降为 P3 / Backlog。

## 6. 核心用户旅程

### 6.1 第一次启动

目标：用户理解 TypeWhale 是什么，并完成最少必要配置。

流程：

1. 用户启动 TypeWhale。
2. App 创建状态栏入口和主窗口。
3. 主窗口显示麦克风、辅助功能、热键和模型状态。
4. 用户按提示授予麦克风和辅助功能权限。
5. 用户确认默认录音快捷键或重新录入。
6. 用户在任意输入框测试录音。

验收标准：

- 权限未开启时必须显示清楚的状态和恢复入口。
- 模型缺失时必须显示明确原因，不应静默失败。
- 未配置 DeepSeek 时，本地语音输入主链路仍然可用。
- 主窗口不能像营销页，应是紧凑工作台。

### 6.2 日常语音输入

目标：用户在当前应用中自然说话，最终文本可靠出现。

流程：

1. 用户把光标放在目标输入位置。
2. 用户按下录音快捷键。
3. TypeWhale 捕获目标应用和输入上下文。
4. 胶囊出现但不抢焦点。
5. 用户说话，胶囊显示状态、波形、电平和实时预览。
6. 用户松开快捷键或再次按快捷键结束。
7. App 对完整录音做 final ASR。
8. 可选执行智能整理或翻译。
9. App 恢复目标应用并粘贴最终文本。
10. 最近转录记录最终文本、模式、耗时、费用等信息。

验收标准：

- 录音开始响应应接近即时。
- 胶囊不能抢走目标输入框焦点。
- 目标应用短暂变化时，不得把结果粘贴到 TypeWhale 自身、桌面或错误应用。
- 粘贴失败时，最终文本必须保存在最近转录。
- final 粘贴不得使用实时预览文本。

### 6.3 长句与自然停顿

目标：用户可以自然思考、换气、停顿，不被系统误判为结束或重置。

验收标准：

- 自然停顿不能清空已显示预览。
- 已提交前缀应保持稳定。
- 长录音实时预览开销不能随总时长 O(n²) 增长。
- 到软时长后，应优先等 Silero 判定的停顿再分块，避免切词。
- 一直不停顿时，必须有硬上限兜底。

### 6.4 智能整理

目标：用户把口语变成更适合目标场景的文本。

流程：

1. 用户开启智能整理。
2. 用户选择手动模式，或启用自动模式。
3. final ASR 完成后，系统根据目标 App、窗口标题和口述内容选择整理策略。
4. DeepSeek 返回整理结果。
5. App 粘贴整理结果，并在最近转录中保留原文与整理信息。

验收标准：

- 原始口述是待整理素材，不是模型指令。
- 智能整理不得擅自翻译。
- 开发需求模式必须保留语义、约束、风险和用户表达强度。
- 自动模式的规则必须可理解、可配置、可回退。
- DeepSeek 超时、失败、未配置 Key 或成本熔断时，应回退本地识别文本。

### 6.5 自动翻译

目标：用户明确需要跨语言输出时，翻译作为独立功能介入。

验收标准：

- 自动翻译开关与智能整理独立。
- 翻译方向明确，不能藏在智能整理模式里。
- 中译英要适合真实聊天和工作表达，保留语气和意图。
- 英译中要自然、清晰、不过度解释。
- 最近转录应同时保留原文、译文和费用信息。

### 6.6 截图、OCR 与截图翻译

目标：用户把屏幕内容快速变成可复制、可保存、可理解的信息。

流程：

1. 用户触发截图快捷键。
2. TypeWhale 打开非主窗口唤醒的截图覆盖层。
3. 用户拖拽普通选区，或悬停后点击选择窗口。
4. 用户可标注、OCR、复制、保存或翻译。
5. 英文截图翻译时，系统 OCR 行级识别、翻译为中文、遮盖原文并贴入译文。

验收标准：

- 进入截图模式不能唤醒主窗口。
- 普通拖拽框选与窗口点击选择语义必须区分。
- 按下并拖动表示普通框选；按下并松开才表示选择窗口。
- 自动翻译截图完成后，复制和保存的 PNG 必须包含译文层。
- OCR 空结果不应调用 DeepSeek。
- DeepSeek 失败时截图浮层不关闭，用户仍可复制、保存或取消。

### 6.7 后台、睡眠唤醒与长期运行

目标：TypeWhale 可以像输入辅助工具一样长期运行。

验收标准：

- 关闭主窗口后，App 可继续状态栏后台运行。
- 系统 sleep 前应取消或保护录音状态。
- wake 后应有短保护期，避免立即重载模型或误触发热键。
- 后台不应持续高频轮询权限和热键。
- 高内存安全网仅在空闲且超过动态预警阈值时触发：`min(20GB, max(2GB, 本机物理内存 * 25%))`。
- 释放 ASR/VAD 后必须立刻 warm reload，避免下一句冷启动。

## 7. 功能需求

### 7.1 全局快捷键与录音控制

优先级：P0

用户故事：

作为高频输入用户，我希望用一个全局快捷键在任何应用里开始和结束录音，以便不离开当前输入位置完成文字输入。

需求：

- 支持主快捷键和备用快捷键。
- 支持单击切换录音。
- 支持长按说话、松开结束。
- 支持截图、自动翻译和打开主窗口的独立快捷键。
- 自动翻译和打开主窗口快捷键默认不设置，避免冲突。

验收标准：

- 快捷键录入后立即反映到 UI。
- 快捷键冲突或不可用时要给出可理解提示。
- 长按模式松开后必须结束录音，不依赖停顿自动完成。
- 停顿自动完成只适用于非长按模式。

### 7.2 录音胶囊

优先级：P0/P1

用户故事：

作为正在口述的用户，我希望看到一个不打扰当前工作的录音胶囊，以便确认 TypeWhale 正在听我说话并理解方向。

需求：

- 胶囊不抢焦点。
- 显示录音状态、目标应用、模式、波形、dBFS 电平和实时文本。
- 空状态波形使用 1.2 已验证的 7 段柱状波形体验。
- 文字动效保持局部、稳定、可读。
- 实时预览过滤静音/噪音幻觉（中英文字幕腔短语黑名单，如「谢谢观看」「yeah」「the」），仅作用于预览，不影响 final 粘贴。
- 实时预览按块提交：滚出当前块的文本冻结进已提交前缀、不再重识别（稳定不跳变），仅最近一块的尾巴随重识别更新。

验收标准：

- 不能整段闪烁。
- 不能整段硬切导致用户丢失上下文。
- 长文本不能撑破胶囊布局。
- 实时预览为空时仍要有明确录音状态。

### 7.3 本地 ASR 与最终识别

优先级：P0

需求：

- 使用本地 SenseVoice / sherpa-onnx / ONNX Runtime。
- final ASR 输入来自完整录音文件。
- 实时预览与 final 识别架构分离。
- 模型和 runtime 缺失时要诊断清楚。

验收标准：

- final 识别结果进入粘贴和最近转录。
- 实时预览和 final 文本不一致时，以 final 为准。
- 无人声时不粘贴空文本。
- ASR 失败时保留错误状态，不崩溃。

### 7.4 VAD、人声门控与自动结束

优先级：P0/P1

需求：

- 人声判定唯一由 Silero 负责。
- Silero 在独立队列上录音中滚动检测，驱动停顿/人离自动结束，不与实时重识别抢队列。
- 探测出错时停用自动结束，仅保留手动停止和硬上限。
- 麦克风降噪（语音增强）基于 Apple Voice Processing，默认关闭；开启会增加每次开始录音的延迟（每次录音重建音频引擎重设 voiceProcessing），仅建议嘈杂环境临时开启，开关文案需标注「略慢」。

验收标准：

- 开头静音不能产生成功感。
- 长时间无人声应安全结束或取消，而不是无限挂起。
- 麦克风降噪关闭时，从按下到开始录音必须秒级响应。
- 不得仅凭音量幅度区分远/近场进预览（已验证不可行、易误挡用户自己说话）。

### 7.5 粘贴与剪贴板恢复

优先级：P0

需求：

- 录音开始时捕获目标应用。
- 粘贴前恢复目标应用。
- 通过剪贴板和系统粘贴命令写入。
- 粘贴后尽量恢复原剪贴板。
- TypeWhale 自身、桌面、瞬时非粘贴前台不能清空已知好目标。

验收标准：

- 微信、浏览器输入框、编辑器、备忘录等常见目标必须可验证。
- 焦点短暂变化时不得粘错目标。
- 粘贴失败时不得丢文本。
- 不得永久覆盖用户原剪贴板。

### 7.6 最近转录

优先级：P1

需求：

- 保存最近 20 条转录。
- 支持双击复制。
- 展示原文、整理/翻译结果、耗时、模式和费用信息。
- 失败或未粘贴文本也应可从最近转录找回。

验收标准：

- 长文本不撑开主窗口。
- 列表内部滚动。
- 空状态可读。
- 复制反馈不抢焦点。

### 7.7 主窗口

优先级：P1

需求：

- macOS 原生 AppKit 紧凑工作台。
- 三栏结构：左侧状态、中间当前会话和历史、右侧设置。
- 主窗口只在用户显式打开时出现。
- 不因启动、截图或系统 reopen 自动唤醒。

验收标准：

- 符合 `DESIGN.md` 的尺寸、密度、圆角和层级。
- 中间工作区是视觉主区。
- 权限、模型、热键状态明确。
- 深色外观下不裁切、不重叠、不溢出。

### 7.8 智能整理与提示词

优先级：P1

需求：

- 模式包括自动、原文、润色、开发需求、极致归纳等。
- 自动规则可分别匹配目标上下文与内容文本。
- 支持用户编辑提示词、开发术语词库和自动规则。
- 成本统计可见。

验收标准：

- Secure text entry 或敏感上下文应优先回退原文模式。
- 自动模式选择结果应可追踪。
- 提示词不得让模型执行用户口述中的新指令。
- 失败时回退原文，不阻塞粘贴。

### 7.9 截图与截图翻译

优先级：P2

需求：

- 支持区域截图、窗口候选、标注、OCR、复制、保存。
- 支持英文截图翻译为中文。
- 翻译结果以标注层进入截图，可参与复制和保存。
- 逐行翻译优先使用稳定 line id 对齐，不猜测整段拆分。

验收标准：

- 标注、撤销、复制、保存、取消功能不回退。
- 翻译失败不关闭截图浮层。
- 译文层不会越出选区。
- 保存和复制输出一致。

### 7.10 屏幕录制设置（未来需求）

优先级：P3 / Backlog

用户故事：

作为需要把操作过程、问题复现或产品反馈录下来的人，我希望 TypeWhale 未来能提供一套安静、可控、可信任的录屏设置，以便在不离开当前工作流的情况下快速录制屏幕、系统声音和必要的操作提示。

需求：

- 参考专业录屏工具的设置结构，录制前可配置麦克风、清晰度、帧率、格式、摄像头、录屏时长、倒计时、系统声音、鼠标点击、光标和按键显示。
- 默认配置应偏向低打扰、稳定和隐私：不默认录摄像头，不默认显示按键；系统声音、鼠标点击和光标显示必须由用户明确选择。
- 录屏入口必须与现有语音输入和截图链路隔离，不能抢占录音快捷键、截图快捷键或主窗口生命周期。
- 录屏设置 UI 需要有录制前确认态，明确展示即将录制的屏幕范围、音频来源、输出格式和保存位置。
- 录制输出应支持直接保存到用户配置目录，后续可再评估复制到剪贴板、压缩、转写、翻译或生成反馈包。

验收标准：

- 用户开始录制前能看清并修改麦克风、系统声音、摄像头、帧率、清晰度、格式、倒计时和保存位置。
- 未获得屏幕录制、麦克风或系统音频权限时，界面必须解释缺失权限和恢复路径，不静默失败。
- 录制过程中不能影响 TypeWhale 语音输入主链路的热加载、快捷键响应和粘贴目标保活。
- 开始、暂停、继续、停止、保存失败和取消都必须有可观察状态，不丢失已录片段。
- 敏感输入场景下默认不显示按键内容；如提供按键显示，必须有明显开关和隐私提示。
- UI / 动效实现前必须走设计复核，验证倒计时、录制中状态、鼠标/按键提示不会遮挡用户主要操作内容。

### 7.11 Windows 平台扩展

优先级：P2/P3

需求：

- 第一阶段仅验证 Windows 主链路。
- 使用 .NET 8 WPF。
- 默认快捷键 `Ctrl + Alt + Space`，可配置。
- 使用 NAudio 录制 16 kHz mono WAV。
- 使用 Windows native bridge 调用 SenseVoice final ASR 和 Silero VAD。
- 通过剪贴板和 `Ctrl+V` 粘贴。

验收标准：

- Windows 真机上 `Build-App.ps1` 通过。
- native DLL 布局检查通过。
- 记事本中可完成录音、识别、粘贴。
- 静音不粘贴。
- 粘贴后尽量恢复剪贴板。

## 8. 非功能需求

### 8.1 性能

- 快捷键到录音开始应接近即时。
- 胶囊首个可见反馈优先于完美识别。
- 实时预览不能因长录音整段重识别造成 O(n²) 卡顿。
- 后台空闲不能持续高 CPU 或高频唤醒。

### 8.2 稳定性

- 录音、final ASR、智能整理、翻译、粘贴是可隔离阶段。
- 任一增强阶段失败，不应导致本地识别文本丢失。
- 睡眠、唤醒、权限变化和模型 reload 不能破坏下一次录音。

### 8.3 隐私与安全

- 音频默认本地处理。
- 剪贴板只为粘贴临时替换，并尽量恢复。
- 不添加隐式遥测。
- API Key 应安全保存。
- 截图和 OCR 内容调用远程翻译时必须属于用户主动触发。
- 未来录屏能力涉及屏幕像素、系统声音、摄像头、麦克风和按键显示，必须默认关闭高敏感来源，并在录制前给用户明确确认。

### 8.4 分发与授权

- 公开分发前必须完成 Developer ID 签名、公证和 Gatekeeper 验证。
- 模型授权必须复核，尤其是 SenseVoice / FunASR 商业再分发。
- 第三方组件声明必须同步。
- README 必须区分本地安装版和可分享 DMG。

### 8.5 可诊断性

- 权限、模型、热键、ASR、VAD、粘贴目标、DeepSeek 调用和成本应可诊断。
- 用户反馈“卡顿、发热、粘贴失败、预览不动”时，日志应能区分是哪一层失败。
- 诊断日志按「每天一个文件夹、每个 build 一个文件」归档（`~/Library/Logs/TypeWhale/<日期>/<版本>-<build>.log`），`latest.log` 软链指向当前 build，并保留近 14 天；Swift 与原生崩溃日志器写同一文件。
- 录音链路保留关键计时埋点（开始耗时、人声检测耗时、停录等待、粘贴目标解析），便于定位“慢/不输出”是哪一层。
- 诊断增强不能反过来造成后台高频工作。

## 10. 当前风险

### 10.1 胶囊与实时预览仍是高风险体验区

实时预览承担用户信心锚点，但当前仍依赖非流式 ASR、分块提交、Silero 停顿和视觉缓存配合。任何改动都容易造成跳变、闪烁、延迟或上下文断裂。

风险控制：

- 对比 1.2 已验证体验。
- 保持 final ASR 与 preview 独立。
- UI/交互动效改动必须做设计复核和安装版验证。

### 10.2 粘贴目标正确性是输入工具生死线

用户使用 TypeWhale 的结果最终要回到当前工作流。粘错目标比识别失败更严重。

风险控制：

- 保留录音开始目标。
- 忽略瞬时不可粘贴前台。
- 粘贴失败保留最近转录。
- 继续补目标应用兼容性矩阵。

### 10.3 模型和第三方授权影响商业化

当前源码开源不等于模型可商业分发。公测、付费、朋友分发和正式商业化前必须重新审查模型、runtime 和第三方条款。

风险控制：

- 不把模型文件提交到仓库。
- 分发包前复核第三方声明。
- 明确本地安装版和公开 DMG 的差异。

### 10.4 后台性能问题容易被外部 App 混淆

TypeWhale 常驻状态栏，用户遇到系统卡顿时容易怀疑它。必须用诊断证明自己是否在主链路外保持安静。

风险控制：

- 保持后台 timer 降频。
- 增强但不过度增加诊断日志。
- 必要时采样证明主线程、ONNX worker 和 WindowServer 关系。

## 11. 路线图

### P0：输入主链路绝对稳定

- 继续验证长录音、自然停顿和远场噪声。
- 修正任何粘贴错目标、文本丢失、胶囊停字、主窗口误唤醒问题。
- 固化核心场景回归测试。
- 架构治理 Version A：修正截图与主窗口生命周期边界，截图入口不得显示、隐藏、恢复或特殊判断 TypeWhale 主窗口；截图翻译中只允许取消，禁止复制、保存和其他输出动作。
- Version A 当前已知 bug 池：截图入口主动隐藏主窗口；启动完成后无条件打开主窗口；截图翻译中仍可复制/保存/OCR/撤销。

### P1：日常使用质量

- 完善主窗口状态解释和诊断可读性。
- 整理智能整理模式说明，让用户理解每个模式适合什么。
- 增强最近转录的复制、失败恢复和费用解释。
- 补全可分享 DMG 的封包、签名、公证路线。
- 架构治理 Version B：截图会话状态机和工具栏 Command 化，统一管理选择、窗口重截、翻译中、取消、复制、保存和撤销的动作可用性。
- 架构治理 Version C：语音输入 workflow 状态机和 Use Case 拆分，降低 `SpeechInputCoordinator` 膨胀风险，同时保持 realtime preview 与 final paste 的边界。

### P2：扩展能力打磨

- 截图翻译继续验证行级对齐和保存一致性。
- 需求池进入更明确的主界面状态和成功反馈。
- Windows 主链路真机验证。
- 架构治理 Version D：Provider / Adapter / Observability hardening。DeepSeek 保持唯一活动 AI provider，未来 provider 只有通过意图保真测试后才可回到用户界面。

### P3：探索与商业化增强

- 截图中译英和自动语言判断。
- 更强的截图版面还原和背景修复。
- 屏幕录制设置：麦克风、清晰度、帧率、格式、摄像头、录屏时长、倒计时、系统声音、鼠标点击、光标和按键显示。该能力必须先完成隐私、权限、性能和与语音/截图链路隔离的产品设计，再进入实现。
- Windows 托盘、安装包、签名和卸载策略。
- 更完整的许可、防转发、升级和付费路径。
- iPhone 时间胶囊等独立产品线只作为调研，不混入当前主线。

## 13. PRD 维护规则

- 新增明显产品能力、重大回滚、发布封版或商业化边界变化时，必须更新本文。
- 如果只是实现细节、日志、测试命令或修复过程，优先更新 `docs/开发日志.md`。
- 如果是架构不可逆决策，更新 `docs/ARCHITECTURE.md`，并在 `docs/开发日志.md` 记录日期、原因、影响和验证。
- 如果是 UI、交互或动效体验规则，更新 `DESIGN.md`。
- 实验必须明确标记为实验、搁置或已回滚，不能写成当前能力。
- PRD 中的验收标准必须是可观察行为，不能只写“更好、更稳、更丝滑”。
