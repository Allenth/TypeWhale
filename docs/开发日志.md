## 2026-06-25：1.3.58 (Build 266) — 高内存释放后立刻热加载 + 主窗口精简与「偏好设置」窗口

### 背景

- 用户观察到内存掉到 102MB（模型已卸载）后，下一次开口第一句会卡顿。要求：「模型卸载后应该立刻装回去，不要等到再次唤起时再加载。」
- 功能持续增多，主窗口右侧「选项」栏拥挤（"系统"分组接近裁切、左栏留白）。用户采纳上一轮 UI 方案：主窗口=运行台+快捷设置，配置项移入带标签页的「偏好设置」窗口（⌘,）。

### 行动类型

- 优化（内存治理）+ 修改（UI 结构重构）。

### 改动内容

1. ASR 高内存安全网由「释放后保持冷态」改为「flush + 立刻热加载」：
   - `SenseVoiceASR.reload()` 现在先 `releaseCachedResourcesOnQueue()`（destroy recognizer + flush 被撑大的 onnxruntime 内存池），再 `loadRecognizer(.current())` 用全新内存池热加载回来（同一串行队列，顺序保证）。
   - `SpeechInputCoordinator.releaseASRResourcesIfIdle` 调用 `nativeASR.reload()`，并新增 30s 冷却 `asrArenaFlushCooldownSeconds`，防止 flush→reload→flush 抖动。
   - 删除已无用的 `asrResourcesReleasedAfterIdle` 标志。
2. 主窗口右栏从「选项」（6 快捷键 + 智能整理配置 + 翻译 + 系统）精简为「快捷设置」：仅保留按会话高频切换的 整理模式 / 自动翻译 / 翻译方向 + 一个「偏好设置…」按钮。
3. 新增「偏好设置」窗口（⌘,，`AppLifecycleCoordinator.showPreferences`），用 `NSTabView` 分 5 个标签：
   - 快捷键：主/备用/截图/截图备用/自动翻译/唤起主页 共 6 行。
   - 智能整理：范围/提示词/术语/Key/余额（「模式」留在主界面快捷设置与胶囊上切换）。
   - 翻译：翻译提示词（「自动翻译」开关与「方向」留在主界面快捷设置）。
   - 截图：截图保存位置。
   - 系统：胶囊实时预览/停顿自动完成/录音时降低电脑声音/开机自动启动。
   - `MainViewController.makePreferencesViewController()` 懒构建一次内容（控件只能有一个父视图），通过 `onOpenPreferences` 闭包与协调器解耦；应用菜单新增「偏好设置… ⌘,」。

### 结果与验证

- `./native/release_local_build.sh` 通过，安装版 1.3.58 (266)，签名校验有效，App 正常启动。

### 后续影响

- 以后新增设置只在对应标签页多一行；新增功能多一个标签页；主窗口保持「运行台 + 快捷设置」不再溢出。
- AGENTS.md「内存治理策略」已补充：释放后必须立刻热加载（reload = flush + warmUp）+ 30s 冷却。

## 2026-06-25：1.3.44 (Build 252) — 取消空闲定时卸载，识别线程 2→3（响应优先）

### 背景

- 用户反馈实时预览有轻微延迟。排查确认两处来自此前的内存优化：① `num_threads` 4→2 使实时识别变慢；② 90 秒“空闲定时卸载模型”导致久未说话后开口第一句要先重载、明显卡顿。快照间隔 0.8s 是固有延迟，不在本次调整范围。

### 修改

- **取消“空闲定时卸载”**：移除 `Timing.idleASRUnloadSeconds` 与 90 秒定时释放；`scheduleIdleASRUnloadIfPossible()` 改为只在内存达到预警阈值且空闲时释放。模型平时保持热加载，开口第一句不再卡顿。
- 内存治理只保留“高内存释放”这条安全网（`releaseASRResourcesIfMemoryElevated`，每秒检查；仅 ≥ 预警阈值且空闲时释放）。
- 识别线程 `num_threads` 2→3，兼顾实时识别速度与内存。

## 2026-06-25：识别准确率排查与实时预览稳态优化

### 背景

- 用户反馈：语音输入的最终结果准确率和胶囊实时预览准确率都出现下降。
- 重点怀疑点：近期将 SenseVoice / Qwen3-ASR 的模型线程数从 4 调整为 2，虽然速度没有明显劣势，但可能影响识别质量。

### 排查

- 使用 `~/Library/Caches/TypeWhale/Recordings` 中的真实录音，临时编译 2 线程与 4 线程原生 SenseVoice 对照工具。
- 对多段真实录音分别跑 2 线程和 4 线程识别，最终文本逐字一致；差异主要体现在耗时，4 线程通常更快。
- 确认最终识别和实时预览不是同一路径：
  - 最终识别：录音结束后先跑 Silero VAD，再对完整音频识别。
  - 实时预览：录音过程中按固定间隔生成快照，快照通过 VAD 后立即识别并更新胶囊。
- 发现坏样本中存在 SenseVoice 直接输出句首多余标点的情况，例如「，目前的这些功能……」；现有清洗器只去掉空白和系统标记，没有去掉句首模型幻觉标点。
- 实时预览首帧 0.30 秒、刷新间隔 0.50 秒过于激进，容易在短音频、半句话、停顿或轻微噪声中触发模型补全，导致预览冒出随机单字、碎片词或多余标点。

- `RecognitionTextNormalizer` 增加句首标点/符号清理，避免模型输出的开头逗号进入最终结果或实时预览。
- `AudioRecorder.RealtimeTiming` 将实时预览首帧从 0.30 秒调整为 0.80 秒，刷新间隔从 0.50 秒调整为 0.80 秒，降低短片段识别噪声。
- `RecognitionTextNormalizerCheck` 增加句首中文逗号和英文句首省略号的回归样例。

### 结论

- 目前证据不支持“2 线程导致识别文本准确率下降”：同音频下 2 线程和 4 线程输出一致。
- 最终结果准确率的明确问题点是清洗器未处理句首幻觉标点。
- 预览准确率下降主要来自实时快照过早、过频，以及 ASR 对短片段的补全倾向；本次先做稳态调整，后续如仍有噪声，可继续加入更强的预览候选确认机制。

## 2026-06-25：复盘 — DeepSeek 成本异常追查与单实例保护（随 1.3.22 / Build 230 落地）

### 现象

- DeepSeek 后台（DT）今天显示约 371,374 tokens（命中 220,928 + 未命中 150,286 + 输出 160），但本机账本当天只有约 30 次调用、19,701 tokens、¥0.005——后台是本机的约 19 倍。

### 排查思路（按顺序）

1. **先验证本机记账是否漏**：从 `launch.log` 统计 `request_start = request_done = usage = 31`，`usage=missing = 0`，`request_skipped = 0`，每条 token 都 1:1 来自 DeepSeek 响应，最大单条仅 740 token。→ 本机对自己发起的每一次调用都精准记账，没有漏记、重复或偏差。
2. **看后台数字的结构特征**：命中缓存高达 22 万、输出却几乎为 0（160）。这是“反复发送同一个大前缀、几乎不产出文本”的调用模式，单靠本机 31 次小调用不可能产生。
3. **排除/确认消费来源**：先怀疑同一个 Key 被别的工具（编程助手等）共用；用户明确确认只有 TypeWhale 在用这个 Key。于是把视角从“别的应用”转到“别的 TypeWhale 进程”。
4. **查进程层**：`ps` 发现有两个 TypeWhale 进程，其中一个已运行约 17 小时（昨天启动的旧二进制残留）。开发期当天反复重编译并 `open`，磁盘二进制换了，但旧进程仍在内存里跑旧代码——旧版没有成本护栏/日志/新账本，其调用绕过了今天的全部统计。两个进程还都装了全局快捷键监听，会重复响应、重复调用 DeepSeek。

### 根因

- 不是记账不准，也不是别的应用——是**幽灵旧进程**：一个旧版本 TypeWhale 仍在后台运行，持续调用 DeepSeek 且不计入新版账本，制造了后台与本机统计的巨大差额。

### 策略与修复

- **立即止血**：`forceTerminate` 掉旧进程（普通 `kill` 在沙箱外 shell 报 operation not permitted，最终由用户在自己终端/活动监视器结束）。
- **根治——启动时单实例保护**（`TypeWhaleApplication.terminateStaleInstances()`）：
  - 最新启动的实例用 `launchDate` 找出所有更早启动的同 `bundleIdentifier` 实例并 `forceTerminate`，保证“最新二进制接管、最旧者退出”，同时兼顾开发期更新和生产期防多开。
  - 因 `applicationShouldTerminate` 默认返回 `.terminateCancel`（菜单栏常驻、拒绝优雅退出），必须用 `forceTerminate` 而非 `terminate`。
  - 调用点放在 `applicationDidFinishLaunching` 而非 `main()` 早期：早期 `NSWorkspace.runningApplications` 不可靠（实测 others=0 漏判）。
- **验证**：`open -n` 强制开第二实例，日志记录 `single-instance: check others=1 stale=1 → forceTerminate stale pid=...`，最终进程数稳定为 1。

### 复盘教训

- 本机估算与官方后台对不上时，排查顺序应是：① 先证明本机记账完整（start=done=usage、无 missing/skipped）；② 再看后台数字的结构特征推断调用模式；③ 最后查“消费源”，且**进程层（多开 / 旧版残留）是最容易被忽略的消费源**。
- 开发期“反复重编译 + open”极易留下幽灵旧进程；单实例保护一举解决开发与生产两个场景，应作为常驻基础设施。

## 2026-06-25：1.3.20 (Build 228) — 复盘：把会话约定固化为 AGENTS.md，而不是停留在聊天记忆里

### 背景

- 用户提出一个关键协作要求：每次变更后都要自动递增版本号和 build 号、构建、覆盖安装并打开本地 App，以便马上测试。

### 思考沉淀

- 聊天里的约定是短期记忆，容易在切换 session 后消失；仓库里的 `AGENTS.md` 是长期记忆，会被后续 coding agent 读取和继承。
- 对这个项目来说，“每次修改后构建安装并打开”不是偏好，而是测试节奏的一部分：
  - TypeWhale 是桌面 App，很多问题只有安装版、权限、全局快捷键、窗口层级和状态栏运行态下才会暴露。
  - 只做源码修改或只跑编译，不能保证用户马上能测到真实行为。
  - 自动递增版本/build 可以避免用户测试时混淆“现在打开的到底是哪一版”。
- 因此，正确做法是提供一个固定入口 `native/release_local_build.sh`，让后续所有代理都执行同一个动作，而不是每次重新临时拼命令。

### 实现过程中的坑

- 第一次脚本实现里，Perl 读取/替换 Info.plist 文本时对 `</key>` 的 `/` 转义不稳，导致正则解析失败。
- 修复后又遇到 `$1` 后面紧跟数字版本号的问题：Perl 把 `$1` + `1.3.17` 误读成 `$11`，把 `build_native_app.sh` 中的版本行替坏。
- 最终修复为：
  - 读取使用 `m{...}` 分隔符，避免 `/` 转义问题。
  - 替换使用 `${1}`，避免捕获组和版本号数字粘连。
- 第一次自动打开 App 时，刚覆盖安装后的 LaunchServices 返回短暂启动失败；脚本增加 `sleep` 和 `open -n` 重试后，自动打开流程稳定。

### 决策

- `AGENTS.md` 是后续 AI session 的强制入口，不是可选参考。
- 后续只要修改仓库内代码、配置、脚本或文档，默认都要执行：

```bash
./native/release_local_build.sh
```

- 只有用户明确要求“只分析、不修改”或“不要构建/不要打开 App”时，才能跳过。
- 这条规则未来如果要调整，也必须改 `AGENTS.md` 和开发日志，不能只在聊天里口头变更。

### 后续影响

- 后续每次变更都会带来一个新的本地 build，这会让版本号增长更快，但换来的是测试确定性和协作连续性。
- 如果某次只是纯文档修正，也仍会触发本地 build；这是用户确认的项目工作流。
- 后续代理如果发现 `AGENTS.md` 与用户最新指令冲突，应以用户最新明确指令为准，并同步更新 `AGENTS.md`，避免规则漂移。

## 2026-06-25：模拟流式传输实现思路与策略复盘（待实现）

### 背景

- 当前 DeepSeek 智能整理链路仍是非流式请求：`DeepSeekChatRequest.stream = false`，最终识别结束后等待完整整理结果，再一次性更新主窗口实时文本、历史记录和粘贴队列。
- 用户体验上的问题是“整理中”阶段缺少可感知进度：用户已经说完，但界面只显示等待文案，容易误以为卡住。
- 本次记录用于后续复盘和实现，不代表当前生产行为已经变成真实流式 API。

### 目标

- 做的是**模拟流式传输**，不是改 DeepSeek API 为真正 streaming。
- 只改善等待期的体感反馈：让界面像“正在生成/正在整理”一样逐步呈现内容或占位反馈。
- 不增加 DeepSeek 调用次数，不改变 token 计费，不改变最终粘贴文本来源。
- 最终结果仍以完整 API 返回值为准，模拟内容只是临时 UI 状态。

### 实现思路

- 保持 `SmartInputRouter.rewrite` 和 `DeepSeekRewriteEngine.complete` 的现有一次性请求边界，避免把网络层、成本账本、超时补记、翻译合并逻辑一起打散。
- 在 `SpeechInputCoordinator` 进入智能整理等待态时启动一个 UI-only 的“模拟流式任务”：
  - 输入源优先使用 final ASR 原文或“正在整理…”模板，而不是调用模型生成中间内容；
  - 以固定节奏把文本按标点、短句或字符块逐步送给 `controller.realtimeDraft` 和 `popup.updateDraft`；
  - 每次更新前检查 `taskID` / session 是否仍然有效，避免旧任务污染新录音。
- 完整 DeepSeek 结果返回后，立即停止模拟任务，用真实整理结果覆盖 UI，并继续走现有 `submitPasteResult` / `SmartUsageLedgerStore.record` / `PasteCoordinator` 链路。
- 如果整理超时、熔断或失败，停止模拟任务并显示现有 fallback 文案；不得把模拟文本写入剪贴板或历史记录。

### 实现策略

- 新增一个小型展示层组件或 helper，例如 `SimulatedStreamingPresenter`，职责只包括：
  - 接收 `taskID`、初始文本、目标更新闭包；
  - 基于 `Task.sleep` 做节奏控制；
  - 支持 `cancel()`，并在取消后不再回调 UI。
- 推荐分片策略：
  - 中文按 2-5 个字或标点后的短片段推进；
  - 英文按单词或短词组推进；
  - 长文本限制最多展示前若干行，避免主窗口和胶囊反复扩张；
  - 节奏可从 45-80ms/片段起步，真实 API 等待越久可逐步放慢，避免过早“播完”后又空等。
- 状态策略：
  - `recording` 阶段仍显示实时 ASR 原文；
  - `finalizing` 阶段切到模拟整理态；
  - `pasting` 阶段只显示真实最终结果；
  - 主窗口和胶囊使用同一 presenter 输出，避免两处状态不一致。
- 安全策略：
  - 模拟流式绝不进入 `PendingPasteResult`；
  - 不计入最近转录；
  - 不触发 `SmartUsageLedgerStore.record`；
  - 不参与 VAD、自动结束、final ASR、翻译或粘贴顺序判断。

### 风险与验收

- 风险：如果模拟文本看起来像真实模型输出，用户可能误以为已经完成。UI 文案需要明确处于“整理中”，真实结果回来后再显示成功状态。
- 风险：胶囊宽度和当前会话卡片可能因模拟文本持续变化而抖动；应复用现有宽度保持、行数截断和固定高度策略。
- 风险：任务取消不彻底会造成上一段录音的模拟文本写入下一段 UI；必须用 `taskID` 和 cancellation 双重保护。
- 验收：
  - DeepSeek 请求数不变，`stream` 仍为 false；
  - 模拟期间不会粘贴中间内容；
  - 真实结果返回后 UI、历史、剪贴板三者一致；
  - 超时/熔断/失败时显示 fallback，不残留模拟内容；
  - 连续录音、快速取消、切换目标 App 时不串台。

## 2026-06-25：1.3.13 (Build 221) — 成本护栏从“日常节流”改为“异常熔断” + 胶囊 App/模式标签

### 设计复盘：成本护栏的定位（重点沉淀）

- 上一版（1.3.12）为排查“当天 4 元异常”加入了 `SmartRewriteCostGuard`，但初始阈值定得过低：单次输出 400 token、单次输入 3000 token、每日 120 次、每日 ¥1.00。
- 用户提出关键产品判断：**这个软件是用来帮自己工作的，不是用来限制自己工作的。** 复盘后确认初始阈值把“正常重度使用”也卡掉了，属于设计定位错误。
- 核对 DeepSeek 官方定价（已联网核实）：`deepseek-v4-flash` = 缓存命中 $0.0028 / 未命中 $0.14 / 输出 $0.28 每百万 token，与代码一致；账户余额计价为人民币（CNY）。按此单价，正常一次润色（约 600 输入 / 200 输出）≈ ¥0.0008，一天 1000 次也才约 ¥0.8。**结论：4 元异常几乎不可能来自正常听写，更可能是同一个 API Key 被其他工具共用，或护栏上线前花的。**
- 由此确立护栏的正确定位：
  - **成本护栏是“异常熔断”，不是“日常节流”。** 阈值要高到正常重度使用永远碰不到，只在程序失控（死循环狂刷 API 等）时兜底。
  - **可观测性优先于限制。** 真正解决“为什么花钱”的是 usage 日志 + Debug 面板 + 余额窗口的当天统计，而不是把功能卡掉。
  - **触顶必须明示，不能静默降级。** 静默退原文会让用户以为“润色坏了”。
  - **货币口径与账户一致（人民币）。** 单价美元 × 固定汇率 7 折算为人民币展示，与余额同币种便于对照（汇率按 7 写死，标注为估算）。

### 修改（成本部分）

- `SmartRewriteCostGuard` 阈值由“节流”改为“熔断”：单次输入 3000→8000 token；单次输出 400→4000 token（仅为上限，按实际生成计费，避免长口述被截断）；每日 120→1000 次；每日成本 ¥1.00→¥6.00（用户确认的安全网）。

## 2026-06-24：放弃 macOS 12 兼容，恢复 macOS 14 健壮基线

### 背景与决策

- macOS 12 上 App 启动录音后崩溃、反复重启。排查确认崩溃发生在 onnxruntime 首次创建推理会话时（VAD 的 `CreateVoiceActivityDetector` 与 SenseVoice 的 `CreateOfflineRecognizer` 都崩），与符号绑定、模型版本无关。
- 根因：onnxruntime 自 1.17.0 起改用 C++20，官方要求 macOS 部署目标 ≥ 13.3/13.4（参见 microsoft/onnxruntime issue #21033、#24277）。项目曾尝试用 sherpa-onnx 1.10.46 自带的「minos 11」onnxruntime 1.17.1 绕过，dylib 符号表虽干净、能 dlopen，但运行时仍在 macOS 12 上硬崩。唯一剩余路径（退到 onnxruntime ≤1.16.x + 配套旧 sherpa）成本高、把握低，且 onnxruntime 在 Apple Silicon 上另有间歇崩溃的已知 bug（#18632）。
- 结论：macOS 12 在可接受成本与把握下不可解。用户决定放弃 macOS 12，按 macOS 14 要求恢复方案，并保留无关新功能。

## 2026-06-15：技术总监发布前 review 修复

行动类型：代码修复、文档修正、发布风险收敛。

技术总监指出：

- 连续录音时，旧任务可能因为 `latestSubmittedTaskID` 被新任务覆盖而无法进入 VAD、ASR 和粘贴流程。
- 用户目录模型 fallback 存在下载器，但识别器只认可内置模型；安装入口也未闭环。
- 每次录音入口可能重复对 239MB SenseVoice 模型计算 SHA-256，造成快捷键响应延迟。
- 开发计划把“识别任务超时、取消、有限重试”标为完成，但当前原生 ASR bridge 尚未实现这部分能力。
- `SpeechInputState` 已定义但尚未成为协调器的唯一状态来源。
- 项目仍缺少自动化测试基线。

本次已修复：

- `SpeechInputCoordinator` 不再用 `latestSubmittedTaskID` 阻止旧任务进入 VAD、ASR 和粘贴；该字段仅用于判断是否更新界面。
- 识别结果仍通过 `PasteCoordinator` 串行粘贴，保持任务目标应用冻结。
- `SenseVoiceModelManifest.preferredModelDirectory` 恢复用户目录 fallback：优先内置模型，其次 `~/Library/Application Support/TypeWhale/Models/sensevoice-native`。
- 模型安装按钮在缺失和失败状态下重新显示，并连接到 `SenseVoiceModelInstaller.install()`。
- SenseVoice 与 Silero VAD 的 `isInstalled` 增加基于路径、大小和修改时间的校验缓存，文件未变化时不重复计算 SHA-256。
- 复审后补充修复：模型校验缓存由裸静态 Dictionary 改为串行队列保护的 `ModelInstallVerificationCache`，避免主线程、安装器回调和 ASR 队列并发读写造成数据竞争。
- 再次复审后补充修复：
  - `SenseVoiceModelInstaller.refresh()` 改为后台队列执行模型完整校验，完成后回主线程发布状态，避免首次启动主线程同步读取约 239MB 模型。
  - `ModelInstallVerificationCache` 提供单飞 `value(for:compute:)`，并发 cache miss 时只执行一次完整 SHA-256。
  - 模型安装 commit 后调用 `rememberVerifiedInstallation` 写入 manifest 缓存，避免刚安装完成后 UI ready 状态再次完整 hash。
  - 当前缓存仍以路径、大小和修改时间作为性能缓存键；若要把 SHA-256 作为强完整性边界，需要另行设计更严格但更慢的校验策略。
- `开发计划.md` 修正 P0.3 状态：ASR 超时、取消和有限重试仍是待办，不再标为已完成。

仍待处理：

- 将 `SpeechInputState` 接入 `SpeechInputCoordinator`，减少 UUID、布尔值和可选变量的散落状态。
- 为 ASR bridge 增加超时、取消和有限重试策略。
- 建立最小自动化测试基线，优先覆盖快捷键手势、连续录音、模型 manifest 和粘贴恢复。

## 当前产品基线

截至 `1.2.3 (128)`，TypeWhale 的当前产品基线是：

- 本地优先 macOS 语音快捷输入工具。
- Swift/AppKit 原生主程序。
