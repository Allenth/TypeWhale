## 1. P0 — 不做就无法商业发行

### 1.1 代码签名 + 公证（Developer ID Notarization）
- 现状需手动放行，商业产品不可接受。
- 需要：Developer ID 证书签名 → `notarytool` 公证 → `stapler` 装订 → DMG 内分发。
- 影响：用户"双击即开"，杀软/Gatekeeper 信任，是后续一切分发的前提。

### 1.2 自动更新
- 集成 **Sparkle**（macOS 桌面应用事实标准），配 appcast.xml + EdDSA 签名。
- 没有自动更新就无法持续修 bug / 推功能，商业版必备。

### 1.3 授权与付费体系（当前完全没有）
- **授权**：试用期 → License Key 校验或订阅校验 → 设备数限制 → 离线宽限期。
- **支付**：建议先用 **Paddle / Lemon Squeezy**（Merchant of Record，代办全球 VAT/税务），省去自建税务合规；若上 Mac App Store 走 StoreKit。
- **落地点**：新增 `Presentation/License` 真实授权 UI + `Infrastructure/Licensing` 校验层；与[埋点方案](埋点方案.md) §4.4 变现漏斗对齐。

### 1.4 隐私政策 + EULA + 官网/落地页
- 收费与合规的基础设施：Privacy Policy（与埋点采集字段一致）、最终用户许可协议、产品官网/下载页/Changelog。

## 2. P1 — 商业级质量与可观测性

### 2.1 远程崩溃/错误上报与关键任务 bug 分析
- 现有 `CrashReporter` 仅本地落盘。接 **Sentry**（Apple/macOS SDK）聚合崩溃、异常、关键任务错误和性能问题，建立线上问题可见性。
- Sentry 负责质量问题深挖：堆栈、release/build、breadcrumb、模块、阶段、错误码。
- 与[埋点方案](埋点方案.md)中的产品分析层协同：Sentry 看“为什么坏”，TelemetryDeck/Aptabase 看“坏得多频繁、影响多少用户”。
- 首批必须覆盖三条关键任务：录音到粘贴、截图翻译、模型下载。

### 2.2 CI/CD + 自动化测试
- 流水线：构建 → 单元/UI 测试 → 签名 → 公证 → 打 DMG → 发布 appcast。
- 为录音→转写→粘贴主链路与截图/OCR/翻译补关键路径测试。

### 2.3 API Key 与后端安全
- Key 存储已达标（Keychain ✅）。商业版进一步：DeepSeek 调用经**自有代理后端**，避免用户自带 Key 或客户端密钥暴露；与埋点 `api_secret` 中转决策统一考虑。

### 2.4 模型完整性校验
- 下载的 ONNX 模型校验 hash（防损坏/中间人），失败可重下。

### 2.5 国际化（i18n）
- 建立 `.strings` / String Catalog 资源，抽离现有中英混排文案，支持至少中/英两套。

## 3. P2 — 增长与运营

- 用户引导（onboarding）：首启权限引导 + 首次成功转写（对齐激活漏斗）。
- 帮助文档 + 反馈渠道 + 客服邮箱。
- 官网 SEO、Changelog 页、应用内评分引导。
- 若做中国市场：ICP 备案、合规声明、数据本地化承诺。

---

## 4. 建议推进顺序

```
P0：公证 → 自动更新(Sparkle) → 授权+支付 → 隐私政策/EULA/官网
        ↓（具备可售卖前提后）
P1：Sentry 质量闭环 → TelemetryDeck 产品分析 → CI/CD+测试 → 模型校验 → i18n
        ↓
并行：埋点与质量可观测性方案（见 埋点方案.md，与 P1 同期落地）
        ↓
P2：onboarding → 支持体系 → 增长/SEO
```
