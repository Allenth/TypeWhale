# TypeWhale 运营与商业化启动计划

状态：启动版 v1  
日期：2026-06-30  
关联文档：[商业化路线图](商业化路线图.md) · [埋点方案](埋点方案.md) · [产品需求文档](产品需求文档.md)

本文回答“从今天开始怎么运营、怎么验证付费、怎么走向可售卖”。它不替代商业化路线图；路线图盘点缺口，本文定义执行顺序、对外叙事、早期用户获取和收入验证。

---

## 1. 我的判断

TypeWhale 现在不应直接做大规模公开售卖，也不应继续只在本地打磨功能。正确路径是：

```text
小范围可信 beta 运营 -> 真实高频使用验证 -> 补齐商业发行 P0 -> 付费预售/早鸟 -> 正式稳定版售卖
```

原因：

- 产品主链路已经具备真实使用价值：全局录音、本地 ASR、胶囊反馈、final 粘贴、智能整理、翻译、截图翻译。
- 但商业发行硬前提仍未闭环：Developer ID 公证、自动更新、授权/付费体系、隐私政策/EULA/官网、模型商业再分发授权。
- 语音输入类产品的商业成败不只看功能多，而看用户是否每天自然触发、是否信任它常驻、是否愿意让它进入工作流。
- 现阶段最有价值的运营不是发大广告，而是找到 20-50 个高频文字输入用户，观察他们是否在 3 天内形成习惯。

## 2. 产品定位

一句话：

> TypeWhale 是一个本地优先的 macOS 语音输入与智能文字处理工具，让高频桌面用户在任何应用里用快捷键完成说话、识别、整理、翻译和粘贴。

对外主张：

- 本地优先：语音识别默认在本机完成。
- 不打断当前工作：不抢焦点的录音胶囊，结果回到原输入位置。
- 为真实工作流服务：不是独立转写页，而是常驻桌面的输入生产力工具。
- 可控 AI 增强：智能整理和翻译是用户可配置的增强层，不替代本地输入主链路。

暂不主打：

- “最便宜的语音转文字”。
- “会议转写工具”。
- “系统输入法替代品”。
- “全平台同步云笔记”。

## 3. 首批目标用户

### 3.1 第一优先级：AI 协作与开发工作流用户

画像：

- 每天在 Cursor、Codex、Claude、ChatGPT、Obsidian、飞书、微信、邮件、GitHub issue 中输入大量文字。
- 经常口述需求、bug、复盘、开发计划、提示词和产品判断。
- 能感知“原意保真”“粘贴到当前应用”“整理不要跑偏”的价值。

为什么先打这群人：

- 他们高频、愿意尝鲜、能提供高质量反馈。
- TypeWhale 的智能整理、开发需求模式、最近转录、术语归一化对他们有明显差异化。
- 这群人更能接受早期 beta 的安装门槛，但也会严厉反馈体验退化。

### 3.2 第二优先级：中文高频办公输入用户

画像：

- 微信、飞书、邮件、文档、客服、运营、销售、产品经理。
- 痛点是打字慢、长句难写、想快速把口语变成可发送文本。

进入条件：

- 安装和权限引导必须足够稳。
- 智能整理输出必须自然，不让用户觉得“AI 把我意思改了”。

### 3.3 暂不优先：会议转写和企业团队市场

原因：

- 会议转写会把产品拉向长音频、说话人分离、云同步、团队管理和合规采购，偏离当前最强的“桌面输入”主战场。
- 企业销售需要采购、账号、权限和支持体系，当前还没有足够基础设施。

## 4. 商业模式建议

### 推荐：早期个人版买断 + 后续 Pro 订阅

第一阶段：

- 早鸟买断：人民币 99-149 元，或美元 19-29。
- 包含当前 macOS 个人版、1 年更新、基础智能功能自带用户自己的 DeepSeek Key。
- 目标不是利润最大化，而是验证“用户愿不愿意为这个输入习惯付钱”。

第二阶段：

- Pro 订阅：人民币 15-29 元/月，或美元 4-8/月。
- 适用于托管 AI 额度、自动更新、跨设备授权、更多模型/翻译额度、优先支持。

不建议现在做纯 Freemium：

- 语音输入工具的免费用户可能很多，但早期支持成本和稳定性压力会快速上升。
- 当前真正要验证的是强需求和付费意愿，不是下载量。

不建议现在做重 DRM：

- TypeWhale 是 local-first 工具，不能让授权校验阻塞录音和粘贴。
- 授权应采用试用期、License Key、本地缓存、离线宽限期和低频校验，不在每次输入前联网。

## 5. 商业发行 P0

这些不做，不能对陌生用户公开收费：

1. Developer ID 签名与公证  
   用户必须能正常双击安装打开，不能要求陌生付费用户手动绕过 Gatekeeper。

2. 自动更新  
   macOS 桌面 App 需要持续修复主链路和兼容性问题，建议集成 Sparkle。

3. 授权与支付  
   先走 Paddle 或 Lemon Squeezy 这类 Merchant of Record，降低全球 VAT/税务和支付合规负担。客户端新增授权 UI 与校验层。

4. 隐私政策、EULA、官网/下载页  
   必须解释本地 ASR、远程智能整理/翻译、错误上报、匿名统计、第三方模型和关闭方式。

5. 模型与第三方商业授权确认  
   SenseVoice / FunASR、sherpa-onnx、ONNX Runtime、OCR/翻译依赖的商业分发边界必须确认。未闭环前，不能把模型再分发写成“已商业授权”。

## 6. 运营启动动作

### 第 1 周：准备可信 beta

目标：能让 20 个真实用户安装、使用、反馈，不追求大规模传播。

交付物：

- Beta 下载页或 GitHub Release 页文案。
- 1 分钟产品介绍：它解决什么、怎么用、隐私边界是什么。
- 安装与权限引导文档。
- 反馈入口：邮箱、表单或微信群。
- 反馈模板：场景、输入目标 App、录音时长、是否粘贴成功、输出是否保真、是否愿意继续用。
- 手工 beta 用户清单与状态表。

必须记录的手工指标：

- 安装是否成功。
- 麦克风/辅助功能授权是否成功。
- 第一次成功粘贴耗时。
- 第一天成功输入次数。
- 第三天是否还在使用。
- 用户愿意支付的价格区间。

### 第 2 周：高频场景验证

目标：找到最强的 2-3 个使用场景，而不是平均服务所有人。

重点观察：

- AI 协作口述需求。
- 微信/飞书长消息。
- 邮件和文档段落。
- 中译英表达。
- 截图翻译是否是主卖点还是增强点。

关键问题：

- 用户是否每天自然唤起 TypeWhale？
- 胶囊是否让用户有信心继续说？
- 输出是否足够保留原意？
- 用户是否愿意为了“少打字 + 快速整理”付费？

### 第 3-4 周：付费意愿验证

目标：在正式支付系统上线前先验证购买意愿。

动作：

- 发布早鸟名额：例如 30 个。
- 明确权益：个人版永久使用/1 年更新/优先反馈/早期价格锁定。
- 使用手工收款或等待 MoR 支付系统，不在产品内伪造未完成的购买体验。
- 记录从访问、咨询、下载、激活、连续使用到愿意付费的每一步。

成功信号：

- 20-50 个 beta 用户中，至少 5-10 人连续 3 天使用。
- 至少 3-5 人明确愿意付费或完成早鸟付款。
- 用户反馈集中在可修复体验问题，而不是“我根本不需要这个”。

失败信号：

- 大多数用户安装后不用。
- 用户只把它当新鲜 demo，不进入真实输入流。
- 用户主要要求会议转写、云同步、手机端等偏离当前定位的能力。

## 7. 对外话术草案

### 标题

TypeWhale：在任何 Mac 应用里，用语音完成输入、整理和翻译。

### 副标题

按下快捷键，说出想法。TypeWhale 在本机完成语音识别，把最终文本粘贴回你正在使用的应用；需要时再用 AI 帮你整理成更清楚的表达。

### 三个卖点

- 本地优先的语音输入：音频默认在本机识别，不把语音当成云端转写任务。
- 不打断工作流：胶囊反馈不抢焦点，结果自动回到原输入位置。
- 为 AI 时代的文字工作而做：口述需求、整理表达、中文转英文、截图翻译，都围绕真实桌面输入场景。

### 风险诚实说明

当前公开测试版仍在完善商业发行能力。正式付费版上线前会补齐签名公证、自动更新、授权、隐私政策和第三方授权确认。

## 8. 指标体系

商业化前先看手工指标，遥测上线后再自动化。

激活：

- 下载到首次打开。
- 首次打开到麦克风授权。
- 首次打开到辅助功能授权。
- 首次成功录音到粘贴。

留存：

- D1 是否完成 3 次以上成功输入。
- D3 是否继续使用。
- 一周内是否在两个以上目标 App 使用。

产品质量：

- record_to_paste 成功率。
- ASR 失败率。
- 粘贴失败率。
- 智能整理失败或用户认为跑偏的比例。
- 用户反馈中的“卡顿、闪烁、粘错、误改意思”次数。

商业：

- 下载到咨询。
- 咨询到安装。
- 安装到愿意付费。
- 早鸟转化率。
- 退款/后悔原因。

## 9. 决策边界

可以现在做：

- 小范围 beta 运营。
- 官网/落地页文案。
- 隐私政策和 EULA 草案。
- 公证、自动更新、授权技术方案。
- 早鸟付费意愿验证。
- 手工用户访谈和反馈整理。

不要现在做：

- 大规模广告投放。
- 企业团队采购。
- 强 DRM。
- 未确认授权前宣称模型可商业再分发。
- 为了卖点堆功能而牺牲主输入链路。
- 把会议转写、云文档、手机端拉进当前商业主线。

## 10. 30 天执行节奏

### 0-7 天

- 定稿 beta 对外文案、安装说明、反馈入口。
- 选 20 个种子用户，优先 AI 协作/开发/产品/运营高频输入人群。
- 开始手工记录激活和 D1 使用情况。
- 同步推进 Developer ID、公证、Sparkle、授权方案调研。

### 8-14 天

- 汇总前 20 个用户反馈，按 P0/P1 分类。
- 修复阻断连续使用的问题。
- 定稿隐私政策和 EULA 草案。
- 确认模型商业再分发路径：取得授权、改为用户自备模型、或替换模型。

### 15-21 天

- 扩展到 50 个 beta 用户。
- 发布早鸟权益说明。
- 确认 Paddle / Lemon Squeezy / StoreKit 取舍。
- 授权 UI 和本地 License 缓存进入开发排期。

### 22-30 天

- 判断是否进入付费 beta：
  - 如果 D3 留存和付费意愿成立，开始补齐 P0 发行基础设施。
  - 如果不成立，回到最强场景重新收窄定位，而不是盲目加功能。

## 11. 外部依据

- Apple 官方公证流程：Developer ID 分发需要 notarization 和 stapling，使用 `notarytool` 提交公证。
- Sparkle 是 macOS App 常用自动更新框架，支持 appcast 与 EdDSA 签名。
- Sentry Apple SDK 覆盖 macOS 崩溃、异常和性能问题分析。
- TelemetryDeck 提供面向 Apple 平台的隐私友好分析。
- Paddle / Lemon Squeezy 属于 Merchant of Record 方向，适合早期独立软件降低税务和支付合规复杂度；最终仍需以当时账户准入、费率和地区政策为准。
