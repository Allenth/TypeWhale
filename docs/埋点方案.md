# TypeWhale 数据埋点与质量可观测性方案

状态：决策版 v2 · 日期：2026-06-29 · 适用版本：1.5.x 起

本文定义 TypeWhale 的产品使用分析、崩溃上报、关键任务错误分析和隐私合规边界。结论：**不再以 GA4 Measurement Protocol 作为主埋点方案**；TypeWhale 采用“双层可观测性”：

- **Sentry**：负责崩溃、异常、关键任务错误、性能问题与线上 bug 分析。
- **TelemetryDeck**：负责匿名产品使用流程统计、激活漏斗、留存、功能渗透和变现漏斗。
- **Aptabase**：作为隐私友好、开源/可自托管的产品分析备选。

核心原则不变：TypeWhale 是 local-first / 隐私优先的 macOS 输入工具，埋点只能回答“发生了什么、耗时多少、结果如何”，不能上传用户内容。

---

## 1. 现状与约束

- 当前代码库无远程遥测/分析代码。
- `Infrastructure/Diagnostics/CrashReporter` 为纯本地实现，无远程聚合。
- 现有诊断日志按「每天一个文件夹、每个 build 一个文件」归档，适合本机排查，但无法形成线上问题趋势。
- API Key 已用 Keychain 保存，安全基线良好。
- 设置项统一走 `AppSettingsStore` / `UserDefaults` 模式。
- 产品需求文档已明确：不添加隐式遥测；音频默认本地处理；截图和 OCR 内容调用远程翻译必须由用户主动触发。

红线：

- 不上传识别文本、剪贴板内容、截图像素、音频、API Key、用户文件路径。
- 不使用设备指纹、序列号、邮箱、手机号、真实姓名作为用户标识。
- 不让上报失败影响录音、ASR、粘贴、截图、OCR、翻译主链路。
- 不在用户未被告知且无法关闭的情况下开启远程统计。

## 2. 平台决策

### 2.1 推荐组合：Sentry + TelemetryDeck

| 需求 | 推荐平台 | 原因 |
|---|---|---|
| 崩溃日志 | Sentry | Apple/macOS SDK 支持自动错误、异常、崩溃和性能问题；崩溃报告可落盘并在下次启动发送，适合桌面 App。 |
| 关键任务 bug 分析 | Sentry | 支持 issue grouping、breadcrumb、tag、context、release/build 维度，适合分析“录音失败、粘贴失败、截图失败、DeepSeek 失败”。 |
| 使用流程统计 | TelemetryDeck | Swift/macOS SDK，定位为隐私友好 App Analytics，适合 local-first 品牌和 Apple 平台应用。 |
| 激活/留存/功能渗透 | TelemetryDeck | 用匿名事件看权限漏斗、首次成功转写、核心功能频次、截图/OCR/翻译使用率。 |
| 自托管/开源替代 | Aptabase | 隐私优先、开源、可自托管，适合作为 TelemetryDeck 替代方案。 |

### 2.2 GA4 Measurement Protocol 降级为备选

旧草案选择 GA4 Measurement Protocol 的理由是“HTTP 直传、零 SDK”。复核后不建议作为主方案：

- Google 官方将 Measurement Protocol 定位为增强自动采集，而不是替代完整客户端采集。
- GA4 MP 不提供 macOS 原生崩溃堆栈、符号化、issue grouping、breadcrumb 和任务上下文。
- 对 TypeWhale 来说，GA4 更适合官网/落地页/广告来源统计，不适合 App 内关键任务 bug 分析。
- 若未来需要统一营销分析，可在官网使用 GA4；App 内仍应保持 Sentry + TelemetryDeck/Aptabase。

### 2.3 不推荐路径

| 方案 | 判断 |
|---|---|
| 只用 GA4 | 不适合崩溃和关键任务 bug 分析；会把质量问题误塞进营销分析工具。 |
| 只用 Sentry | 能看质量问题，但产品漏斗、留存和功能渗透不如专门 App Analytics 清晰。 |
| 只用 PostHog | 产品分析强，但 TypeWhale 当前最急的线上质量闭环仍更适合 Sentry；除非明确要自托管、feature flag 和实验平台一体化。 |
| Firebase Analytics SDK | macOS 支持和依赖形态不适合轻量 local-first 桌面工具。 |

## 3. 客户端架构

新增模块建议：

```text
native/Sources/Infrastructure/Observability/
├── ObservabilityClient.swift        // 对业务层门面：track / captureError / breadcrumb
├── AnalyticsEvent.swift             // 产品分析事件枚举 + 参数模型
├── QualityEvent.swift               // 错误/崩溃/关键任务上下文模型
├── AnalyticsTransport.swift         // TelemetryDeck / Aptabase / noop
├── SentryQualityTransport.swift     // Sentry 初始化、tag、breadcrumb、error capture
├── ObservabilityConsentStore.swift  // 用户同意、开关、队列清空
├── ObservabilityQueue.swift         // 产品分析离线队列、批量、重试
└── ObservabilityRedactor.swift      // 参数白名单 + 内容脱敏
```

设计要点：

1. **统一门面，分层上报**：业务代码只调用 `ObservabilityClient`；产品事件送 TelemetryDeck/Aptabase，错误和崩溃送 Sentry。
2. **匿名 ID**：首次启动生成随机 UUID，存 `UserDefaults` 或 Keychain；禁止设备指纹。
3. **白名单参数**：事件参数必须由 enum/schema 定义，默认拒绝自由字符串。
4. **离线队列**：产品分析事件先入本地队列，断网不阻塞；关闭统计时清空。
5. **Sentry 脱敏**：配置 `beforeSend`，剔除可能含文本、路径、截图、音频、剪贴板和 Key 的字段。
6. **低优先级线程**：上报处理不得在录音、ASR、粘贴热路径做重活。
7. **环境隔离**：开发、本地测试、公开 beta、正式发布使用不同 DSN/App ID 或 tag。

## 4. 事件字典

所有事件隐式携带：

- `app_version`
- `build`
- `os_version`
- `locale`
- `release_channel=local|beta|stable`
- `anonymous_user_id`
- `session_id`

禁止携带：

- 识别文本
- 原始音频
- 截图像素
- OCR 文本
- 剪贴板内容
- API Key
- 具体文件路径
- 目标输入框内容

### 4.1 激活漏斗

| 事件 | 平台 | 触发时机 | 参数 |
|---|---|---|---|
| `app_launch` | Analytics | 启动完成 | `is_first_launch`, `launch_source` |
| `permission_prompt` | Analytics | 弹出权限请求 | `type=mic|accessibility|screen_recording` |
| `permission_result` | Analytics | 权限结果变化 | `type`, `result=granted|denied|unknown` |
| `model_download_start` | Analytics | 开始下载模型 | `model` |
| `model_download_result` | Analytics + Sentry error on fail | 下载结束 | `model`, `result`, `duration_ms`, `fail_reason` |
| `first_successful_transcription` | Analytics | 首次成功转写 | `model`, `backend` |

### 4.2 核心动作

| 事件 | 平台 | 参数 |
|---|---|---|
| `record_start` | Analytics breadcrumb | `mode=hold|toggle`, `input_device_mode=system|manual` |
| `record_complete` | Analytics breadcrumb | `duration_ms`, `cancelled`, `reason` |
| `asr_complete` | Analytics breadcrumb | `char_count_bucket`, `duration_ms`, `model`, `backend` |
| `asr_fail` | Analytics + Sentry error | `fail_reason`, `stage=vad|asr|postprocess` |
| `paste_result` | Analytics + Sentry error on fail | `result`, `fallback_used`, `target_app_category` |
| `smart_rewrite_run` | Analytics + Sentry error on fail | `has_api_key`, `result`, `duration_ms`, `token_bucket` |
| `screenshot_capture` | Analytics breadcrumb | `type=region|window`, `result` |
| `ocr_run` | Analytics + Sentry error on fail | `result`, `duration_ms`, `line_count_bucket` |
| `screenshot_translate` | Analytics + Sentry error on fail | `direction`, `result`, `duration_ms`, `line_count_bucket` |

### 4.3 质量与性能

| 事件 | 平台 | 参数 |
|---|---|---|
| `crash` | Sentry | 自动崩溃报告；额外 tag：`app_version`, `build`, `release_channel` |
| `error` | Sentry | `module`, `code`, `fatal=false`, `task` |
| `critical_task_failed` | Sentry + Analytics aggregate | `task=record_to_paste|screenshot_translate|model_download`, `stage`, `code` |
| `perf_asr_latency` | Analytics | `first_preview_ms_bucket`, `total_ms_bucket` |
| `perf_paste_latency` | Analytics | `total_ms_bucket`, `fallback_used` |

### 4.4 变现漏斗

付费体系上线后启用：

- `paywall_view`
- `trial_start`
- `purchase_success`
- `license_activate`
- `license_check_failed`

## 5. Sentry 事件上下文规范

Sentry 只记录排查 bug 必须的结构化上下文：

- release：`1.5.x (build)`
- tag：`module`, `task`, `stage`, `release_channel`, `os_version`
- breadcrumb：最近 20 个关键动作，例如 `record_start`、`asr_complete`、`paste_result`
- extra：只允许枚举、耗时、计数、bucket，不允许用户内容

示例：

```text
module=paste
task=record_to_paste
stage=restore_clipboard
code=pasteboard_write_failed
duration_ms_bucket=500_1000
fallback_used=true
```

Sentry 禁止字段：

- raw transcript
- prompt
- OCR text
- screenshot image
- clipboard value
- file path
- API key
- account/license email

## 6. 隐私合规

必须满足：

1. **显式同意**：设置页提供“匿名使用统计与错误报告”开关；首次启动或首次进入设置时说明用途。
2. **默认策略**：商业公开版建议默认 opt-in；若目标市场包含欧盟，优先 opt-in。内部测试版可单独配置。
3. **分项开关**：至少拆成两个逻辑开关：
   - 匿名使用统计
   - 崩溃与错误报告
4. **关闭即停止**：关闭后停止新上报并清空本地产品分析队列。
5. **零内容采集**：任何用户生成内容不得进入事件、breadcrumb、extra、message。
6. **隐私政策**：公开 Privacy Policy，写清楚 Sentry、TelemetryDeck/Aptabase、采集字段、关闭方式、保留期。
7. **PrivacyInfo.xcprivacy**：与实际采集字段一致。
8. **本地日志独立**：`~/Library/Logs/TypeWhale` 仍作为本地诊断，不自动上传完整日志。

## 7. 落地计划

### P0：决策与隐私基线

- 确认平台：Sentry + TelemetryDeck；Aptabase 作为备选。
- 写隐私政策草案。
- 设计设置页开关和首启说明。
- 定义事件 schema 和脱敏白名单。

### P1：Sentry 质量闭环

- 接入 Sentry Apple/macOS SDK。
- 配置 release/build/environment。
- 接入 `beforeSend` 脱敏。
- 对 `record_to_paste`、`screenshot_translate`、`model_download` 三个关键任务加 breadcrumb 和 error capture。
- 保留现有本地 `CrashReporter`，作为 Sentry 之外的本机兜底。

### P2：产品分析闭环

- 接入 TelemetryDeck Swift SDK，或在最终评估后改用 Aptabase。
- 建 `ObservabilityClient` 与 `AnalyticsEvent`。
- 接入激活漏斗和核心动作低频事件。
- 建 dashboard：激活、留存、核心功能渗透、失败率、关键耗时 P50/P95。

### P3：商业化漏斗

- 授权/试用/付费上线时加入 license 与 paywall 事件。
- 与支付服务商事件对账，避免重复统计收入。

## 8. 验收标准

- [ ] 关闭“匿名使用统计”后，抓包确认无产品分析上报且本地队列清空。
- [ ] 关闭“崩溃与错误报告”后，Sentry 不发送新事件。
- [ ] 断网录入、恢复网络后产品分析事件不丢、不重复、不阻塞主链路。
- [ ] Sentry 测试错误能按 release/build/environment 聚合。
- [ ] Sentry 事件不包含识别文本、剪贴板、截图、OCR 文本、API Key、文件路径。
- [ ] `record_to_paste` 失败能看到 stage/code/duration/fallback breadcrumb。
- [ ] `screenshot_translate` 失败能看到 OCR/translation/layout 阶段，但看不到 OCR 文本。
- [ ] `PrivacyInfo.xcprivacy` 与隐私政策和实际字段一致。

## 10. 官方依据

- Sentry Apple/macOS SDK：支持 macOS 错误、异常、崩溃与性能问题；崩溃报告可落盘并在下次启动发送。
- TelemetryDeck：提供 Swift SDK，支持 iOS、macOS、watchOS、tvOS 等 Apple 平台，定位为隐私友好 App Analytics。
- Aptabase：开源、隐私优先、可云端或自托管，面向移动、桌面和 Web App。
- Google Analytics Measurement Protocol：官方定位是增强自动采集，不是完整客户端采集替代方案。
